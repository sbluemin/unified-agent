# `unified-agent` TUI/CLI 스타일링 라이브러리 도입 종합 보고서

## 1. 개요
현재 `unified-agent`의 진입점인 `@src/cli.ts`는 `isTTY` 여부에 따라 간단한 ANSI escape 코드를 직접 구현하여 사용하고 있습니다(`c.bold`, `c.cyan`, `c.green` 등). 
본 보고서는 2024년 최신 Node.js CLI/TUI 트렌드를 바탕으로, 현재의 수동 구현 방식을 대체하고 향후 확장성을 고려할 수 있는 최적의 오픈소스 라이브러리를 조사하고 분석한 결과입니다. (웹 리서치 및 Codex AI 코드 분석 결과 종합)

---

## 2. 최신 Node.js CLI/TUI 라이브러리 동향 (2024)

최근 CLI 생태계는 무거운 전통적 프레임워크에서 벗어나, 용량이 작으면서도 심미성이 뛰어난 도구들로 이동하고 있습니다.

*   **인터랙티브 UI (트렌드 리더)**
    *   **Clack (`@clack/prompts`)**: 현재 가장 주목받는 라이브러리로, 매우 미니멀하고 세련된 UI를 제공합니다. 다른 라이브러리보다 가벼우며 스피너, 프로그레스 바 등을 우아하게 렌더링합니다.
    *   **Inquirer (`@inquirer/prompts`)**: 오랫동안 표준이었으며 최근 현대적으로 개선되었으나, 여전히 다소 무거운 편입니다.
*   **복잡한 레이아웃 (TUI 프레임워크)**
    *   **Ink**: React를 사용하여 터미널 UI를 빌드하는 강력한 도구입니다. 복잡한 대시보드 구조에 적합합니다.
*   **경량 텍스트 스타일링 (Chalk 대체제)**
    *   **picocolors**: Zero-dependency, CJS/ESM 지원, 빠른 속도를 자랑하는 최신 텍스트 스타일링 유틸리티입니다.
    *   **yoctocolors**: `chalk` 제작자가 만든 초경량 스타일링 도구이나, ESM/CJS 패키지가 분리되어 있습니다.

## 3. Codex의 프로젝트 구조 맞춤 분석

Codex를 통해 현재 `unified-agent`의 구조(`@src/cli.ts`, `package.json`, 파이프라인 처리 방식)를 분석한 결과, **현재의 원샷(One-shot) 스트리밍 구조에서는 거대한 UI 프레임워크보다 가벼운 스타일링 유틸리티 도입이 가장 적합**하다는 결론을 얻었습니다.

### ❌ 도입 비추천 (현재 구조 기준 오버엔지니어링)
*   **Ink**: CLI 전체를 React 컴포넌트 트리로 재설계해야 하므로, 단순 문자열 스트리밍을 수행하는 현재 앱에는 지나치게 무겁습니다.
*   **Inquirer / Clack**: 이들은 기본적으로 "대화형 프롬프트(Interactive Prompts)" 라이브러리입니다. `unified-agent`는 `stdin` 파이프와 인자를 통한 원샷 실행이 1급 경로로 다뤄지고 있으므로, 당장 수동 ANSI 헬퍼 교체 용도로는 과도합니다. 추가로 Inquirer는 Node `>=20.12.0` 이상을 권장하여 프로젝트의 `>=18` 조건과 충돌할 수 있습니다.

### ✅ 강력 추천: `picocolors`
*   **적합성 요인**:
    *   현재 코드는 문자열 스타일 함수만 구현하고 있으며, 실제 동작은 `stdout` 스트리밍과 얇은 상태 출력(`stderr`)이 전부입니다.
    *   `picocolors`는 공식적으로 **Zero-dependency, CJS/ESM 동시 지원, TS 타입 포함**을 제공하며 패키지 의존성을 극도로 낮게 유지하는 현재 프로젝트 방향성과 완벽히 일치합니다.
    *   현재의 `isTTY`, `isErrTTY` 구조를 `createColors(enabled)` API를 통해 1:1로 가장 매끄럽게 교체할 수 있습니다.
    *   현재 수동 구현이 놓치고 있는 `NO_COLOR` / `FORCE_COLOR` 환경변수 처리 문제를 자동으로 해결해 주어 테스트 안정성을 높입니다.

*   **차선책 (`yoctocolors`)**: 매우 가볍고 빠르나 CJS/ESM이 분리되어 있어 듀얼 출력을 지원하는 현 프로젝트 설정에서는 덜 매끄러울 수 있습니다.

## 4. 최종 도입 전략 (Action Plan)

1.  **1단계 (단기 교체 - 스타일링 안정화)**
    *   `picocolors` 라이브러리를 도입하여 `@src/cli.ts`의 `c.bold`, `c.cyan` 등의 수동 ANSI 객체를 교체합니다.
    *   `NO_COLOR=1` 등 환경변수를 통한 스타일 끄기 기능의 신뢰성을 확보합니다.
2.  **2단계 (중장기 확장 - UX 고도화)**
    *   만약 향후 `unified-agent init` (설정 마법사)이나 대화형 모델 선택 기능 등 **Interactive 기능**이 추가된다면, 그때 **`@clack/prompts`**를 부분적으로 도입하여 세련된 사용자 경험을 제공하는 것을 권장합니다.

## 5. 현업 개발자 동향 및 선호도 (Perplexity 분석)

2024년 기준 현업 개발자들의 라이브러리 도입 트렌드는 **"목적과 규모에 따른 명확한 분리"**로 요약됩니다.

### 상황별 선호도 요약
*   **간단한 프롬프트 & 레거시 유지보수**: `Inquirer` (가장 안정적이고 검증된 기본값)
*   **세련된 설치 마법사 & 최신 CLI 도구**: `Clack` (모던 UX/UI 트렌드 리더)
*   **상태가 복잡한 TUI 및 대시보드 (AI CLI 등)**: `Ink` (React 기반 생태계 활용)
*   **단순 텍스트 강조 및 고성능 요구**: `picocolors` (가장 추천됨), `yoctocolors`

### 라이브러리별 현업 평가 (장단점)

#### 1. Clack (@clack/prompts)
*   **장점**: 스피너, 작업 로그, 자동 완성 등 기본 스타일이 매우 깔끔하며, ESM/TypeScript 환경에서 "5분 만에 예쁜 CLI 만들기" 예제로 각광받고 있습니다. v1.0 이후 프로덕션 안정성도 크게 높아졌습니다.
*   **단점**: 문서가 다소 부족해 복잡한 로직을 구현할 때 한계가 있으며, CJS 프로젝트에서는 도입이 까다롭습니다.
*   **현업 인식**: 새로운 프로젝트나 서브 커맨드를 작성할 때 기존의 Inquirer 대신 우선적으로 고려되는 **"신흥 강자"**입니다.

#### 2. Inquirer
*   **장점**: npm 주간 수천만 다운로드를 기록하는 베테랑 라이브러리로, 풍부한 플러그인 생태계와 방대한 문서, 안정성을 자랑합니다. 복잡한 워크플로우 지원에 강합니다.
*   **단점**: API가 약간 구식(CJS 기반에서 발전)이라는 인식이 있고, 기본 테마가 요즘 기준으로는 투박합니다.
*   **현업 인식**: 새로 배우기보다는 "이미 알고 있어서 쓰는" 경우가 많으며, 보수적인 사내 도구나 거대한 기존 프로젝트 유지보수용으로 계속 채택됩니다.

#### 3. Ink
*   **장점**: React의 VDOM과 Hooks 생태계를 터미널로 가져와 패널, 사이드바 등 복잡한 레이아웃을 구성하는 데 압도적으로 유리합니다. (Gatsby, GitHub Copilot 등 대형 도구에서 사용)
*   **단점**: 단순한 CLI에는 심각한 오버엔지니어링이며, 무거운 TUI 앱에서는 VDOM 리렌더링으로 인한 성능/메모리 이슈가 종종 지적됩니다.
*   **현업 인식**: React 기반 프론트엔드 팀이 터미널 앱(예: AI 에이전트, 모니터링 도구)을 만들 때 절대적인 지지를 받습니다.

#### 4. picocolors & yoctocolors (스타일링 유틸)
*   **picocolors 장점**: "Chalk의 가장 완벽한 경량 대체재". 주간 8,000만 다운로드를 돌파했으며, Zero-dependency로 아주 작고 빠르며 CJS/ESM을 모두 잘 지원합니다.
*   **yoctocolors 장점**: Chalk 원작자가 만든 가장 작고 빠른 라이브러리. 극단적인 번들 사이즈 최적화가 필요한 툴체인(린터 등)에서 선호됩니다.
*   **현업 인식**: 대부분의 개발자가 무거운 Chalk 대신 이 둘 중 하나를 선택하는 추세이며, 범용적인 호환성과 생태계 면에서 **`picocolors`가 사실상 표준(de-facto)**으로 자리 잡고 있습니다.
